# 抗重放 (Anti-Replay) 与填充 (Padding)

Prisma 包含两种防御网络层攻击的机制：滑动窗口 (Sliding Window) 抗重放机制和握手 (Handshake) 消息的随机填充。

## 抗重放保护 (Anti-Replay Protection)

### 滑动窗口 (Sliding Window)

Prisma 使用 1024 位滑动窗口来检测重放或重排序的 nonce 计数器值。该窗口实现为 16 个 `u64` 字的位图（共 1024 位）。

### 工作原理

- 窗口跟踪一个 `base` 计数器和相对于该基准的 1024 位已见计数器位图
- 每个传入帧的 nonce 计数器会与窗口进行检查：
  - **低于 `base`**：作为过旧拒绝（比最高已见值落后超过 1024 帧）
  - **在窗口内且已标记**：作为重放拒绝
  - **在窗口内且未见**：接受并在位图中标记
  - **超出窗口**：窗口向前推进，旧条目被丢弃，新计数器被接受

### 特性

- 允许最多 1024 帧的乱序传递
- 检测窗口内的精确重放
- 随着计数器推进自动驱逐过期条目
- 零堆分配——固定大小 `[u64; 16]` 数组

### 错误处理

当检测到重放时，该帧会被拒绝并返回 `ReplayDetected` 错误，包含违规的计数器值。连接可以继续处理后续有效帧。

## 随机填充 (Random Padding)

### 握手填充 (Handshake Padding)

`ClientInit` (v3) / `ClientHello` (v1/v2) 和 `ServerInit` / `ServerHello` 消息都包含随机填充以抵抗流量指纹识别。填充格式为：

```
[padding_len:2 字节 大端][random_bytes:padding_len]
```

- 填充长度从 0 到 `max_size` 字节（最多 256）随机选择
- 填充字节使用密码学安全的随机数据填充
- 接收方通过读取 2 字节长度前缀来剥离填充

### 逐帧填充 (Per-Frame Padding) (v2/v3)

在协议 v2 和 v3 中，当设置 `FLAG_PADDED` 时，每个数据帧都包含随机填充。填充帧格式为：

```
v3: [command:1][flags:2 小端][stream_id:4][payload_len:2][payload:可变][padding:可变]
v2: [command:1][flags:1][stream_id:4][payload_len:2][payload:可变][padding:可变]
```

- 填充范围（最小/最大字节数）在握手期间通过 `ServerInit`（v3）或 `ServerAccept`（v2）消息协商
- 默认范围：0-256 字节
- 可通过服务端配置中的 `[padding]` 部分配置：

```toml
[padding]
min = 0     # 每帧最小填充字节数
max = 256   # 每帧最大填充字节数
```

- 每个帧独立地在协商范围内随机选择填充长度
- `payload_len` 字段使接收方能够区分有效载荷和填充
- 这可以防止将帧大小与应用程序数据大小关联的流量分析

### 为什么填充很重要

如果没有填充，握手消息的大小是可预测的，因为它基于固定大小的字段（公钥、时间戳）。观察者可以通过匹配这些大小来指纹识别 PrismaVeil 流量。随机填充使每次握手的总大小都不同。

逐帧填充（v2）更进一步：如果没有它，数据帧大小与应用程序数据大小直接相关，使得统计流量分析成为可能。通过随机化的逐帧填充，帧大小与数据大小之间的关系被模糊化。

## 协议常量

| 常量 | 值 | 描述 |
|------|-----|------|
| `MAX_PADDING_SIZE` | 256 | 最大握手填充大小 |
| `DEFAULT_PADDING_RANGE` | 0-256 | 默认逐帧填充范围 |
| 窗口大小 | 1024 位 | 抗重放窗口容量 |
| 位图字数 | 16 | 位图中 `u64` 字的数量 |
